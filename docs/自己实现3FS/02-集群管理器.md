# 集群管理器

## 这个模块解决什么问题

`mgmtd` 的职责不是做数据存储，而是做控制面协调。自己实现一个 3FS 风格系统时，集群管理器至少要负责：

- 节点注册和成员发现。
- 基础租约与心跳。
- 路由信息分发。
- 目标设备与 chain 的全局视图。
- 配置模板下发。

如果没有一个明确的控制面，元数据服务和存储服务迟早会彼此耦合。

## MVP 怎么实现

最小版建议做成单主进程，状态放在一个简单持久化存储中。

### 最小状态

- `NodeInfo`
  节点 id、地址、节点类型、最后心跳时间。
- `RoutingInfo`
  meta 列表、storage 列表、chain 到 target 的映射。
- `ConfigInfo`
  按节点类型保存默认配置模板。

### 最小接口

- `register_node`
- `heartbeat`
- `get_routing_info`
- `get_config`

### 最小租约模型

建议一开始就引入租约而不是单纯“在线/离线”：

- 节点注册后获得一个 lease。
- 心跳本质上是续租。
- 超时未续租则视作失效。

这样后面加主备、自动摘链、恢复状态机会更自然。

## 为什么真实工程会更复杂

一旦进入多副本和恢复场景，mgmtd 必须承担更多职责：

- 跟踪 chain version。
- 在 target 离线时调整链表顺序。
- 判断 returning target 何时可从 `offline` 进入 `waiting/syncing/serving`。
- 决定哪些配置是模板，哪些是实例态。

所以 mgmtd 不是“服务注册中心”，而更像一个轻量控制平面。

## 当前 3FS 风格值得借鉴的点

### 用统一状态对象管理写路径

mgmtd 的写路径应当显式串行化，避免多个“读改写”操作互相踩状态。一个很实用的方法是：

- 读状态可以并发。
- 所有会修改链表、租约、会话、配置版本的操作共享一把逻辑写锁。

### 路由信息需要带版本

客户端看到的 routing info 不应只是一个地址列表，而应该有 `routing_version` 或 `chain_version`。这样才能解决：

- storage 看到的链版本与客户端不同。
- 恢复中的链结构发生变化。
- 写请求打到了过时 successor。

## 自己实现时推荐的取舍

### 第一版建议

- 单主 mgmtd。
- 路由信息直接由 mgmtd 内存维护，定期快照持久化。
- 配置模板只按 `node_type` 存。

### 第二版再加

- lease 的可信区间。
- routing info 版本号。
- target 状态机。
- 客户端会话追踪。

## 常见坑

### 坑 1：把 mgmtd 做成“纯内存注册中心”

如果 mgmtd 完全无持久化，一旦重启：

- chain 定义丢失。
- target 状态丢失。
- 配置模板丢失。

因此至少要把链定义、模板配置和节点持久态保存下来。

### 坑 2：不把链路状态单独建模

一开始偷懒只存 `node -> targets` 很容易，但后面做复制和恢复时，一定需要：

- `chain`
- `chain table`
- `target public state`
- `target local state`

不提前建模，后面很难补。
