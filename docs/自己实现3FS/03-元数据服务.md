# 元数据服务

## 这个模块解决什么问题

元数据服务负责的是“文件系统语义”，不是数据块传输。它至少要保证：

- 目录树结构正确。
- inode 与名字映射一致。
- rename/remove/create 具有合理的原子语义。
- 文件布局信息可被客户端取走并用于数据访问。

## MVP 怎么实现

### 关键对象

- `Inode`
  文件、目录、符号链接的统一元数据对象。
- `DirEntry`
  父目录下名字到 inode 的映射。
- `Layout`
  文件对应的数据放置描述。

### 最小语义集

第一版建议实现：

- `stat`
- `list`
- `mkdirs`
- `create`
- `open`
- `close`
- `rename`
- `remove`
- `truncate`

### 最小事务边界

建议 rename / create / remove / hardlink 这类操作一定放进单事务。理由很简单：

- 文件系统最大的价值之一是名字空间的一致性。

如果目录项和 inode 更新不是原子的，很快就会出现“看得到名字，找不到 inode”或“inode 存在但目录树不可达”的问题。

## 最小版应该如何存储元数据

最推荐的是：

- 使用一个支持事务的 KV 存储。

如果先做原型，可以用：

- SQLite
- RocksDB + 自己的写事务封装
- FoundationDB 一类现成事务 KV

如果一开始用普通内存 map，也不是不行，但很快会遇到：

- 重启丢状态
- 无法验证并发更新
- 事务语义无法逼真模拟

## 目录树操作怎么想

### `create`

建议拆成两个步骤，但放在一个事务中：

1. 分配新 inode。
2. 在父目录里插入新的 `DirEntry`。

### `rename`

最容易低估的操作之一。正确做法是：

- 同时处理源父目录和目标父目录。
- 处理覆盖目标时的删除语义。
- 防止目录移动到自己的子目录中形成环。

### `remove`

建议先区分：

- `unlink file`
- `rmdir empty dir`
- `rm -r recursive remove`

递归删除第一版可以简单做，但接口上要为后续 GC 保留空间。因为真实大目录删除通常不会同步删完所有块。

## 为什么真实工程会更复杂

### 文件长度不是只靠 meta 自己能知道

真实分布式文件系统里，文件长度往往由数据写入路径决定。meta 只能知道：

- 当前已知长度
- 客户端的 hint

真正的最终长度可能要查询 storage 才能确认。

### 打开/关闭不仅是引用计数

一旦你支持：

- 多客户端并发写
- 延迟 flush
- close 后更新长度

那么 open/close 就会演化成会话管理问题，而不仅是内存里的引用计数。

## 向 3FS 风格演进时该加什么

### 演进 1：批处理同一 inode 上的高频操作

对同一个 inode 的：

- sync
- close
- setAttr
- create child

可以按 inode 聚合后一起执行，减少事务提交和重复查询。

### 演进 2：把布局分配抽成独立组件

文件创建时不应直接把 target 写死在 meta 逻辑里。建议单独做：

- `LayoutAllocator`
- `ChainAllocator`

这样后面从单 target 升级到 chain table 时，不需要重写 create 主逻辑。

## 自己重做时推荐的取舍

- 第一版就让 `Layout` 成为 inode 的一部分。
- 第一版就把 `rename` 当困难点认真做，不要只实现“同目录改名”。
- 递归删除可以先简单，但要把“目录删除”和“块删除”两个概念分开。

## 常见坑

### 坑 1：把文件长度当作单一字段同步更新

真实系统里长度往往有：

- 客户端观察值
- meta 当前值
- storage 实际值

第一版可以简化，但接口不要只留一个“绝对可信长度”的思路。

### 坑 2：创建文件时不记录布局

如果布局完全靠客户端临时决定，后面：

- 恢复
- reread
- query chunk
- truncate

都会变得很难统一。
