# 故障恢复与状态机

## 这个模块解决什么问题

复制本身只解决“正常情况下数据怎么传播”，并不自动解决：

- 节点失效后链怎么调整。
- 节点恢复后如何重新加入。
- 恢复中的节点是否可读可写。

所以恢复一定要配合状态机来做。

## MVP 怎么做

第一版可以接受：

- 节点失败后人工摘除。
- 节点恢复后人工重建。

但即使如此，也建议先定义 target 状态，而不是只用一个布尔值 `alive/dead`。

## 推荐的状态模型

至少建议区分两个维度。

### 公共状态

这是客户端和其他服务可见的状态：

- `serving`
- `waiting`
- `syncing`
- `offline`

### 本地状态

这是 storage 与 mgmtd 内部使用的状态：

- `up_to_date`
- `online`
- `offline`

公共状态决定：

- 客户端能不能读。
- 写路径是否继续向它传播。

本地状态决定：

- 它是否真的完成恢复。
- mgmtd 下一步应该把它推进到哪里。

## 节点故障后应该发生什么

推荐顺序：

1. mgmtd 检测 lease 过期。
2. 标记相关 target 为 `offline`。
3. 把它们移到 chain 末尾或从可服务链路中摘出。
4. 发布新的 chain version。
5. 客户端和 storage 在下一次路由刷新后切换到新链。

## 节点回来后应该发生什么

推荐让 returning target 走一个明确阶段：

1. `offline -> waiting`
2. `waiting -> syncing`
3. `syncing -> serving`

### `waiting`

说明它回来了，但尚未开始真正恢复。

### `syncing`

说明它正在从前驱或其他副本拉取所需 chunk。

### `serving`

说明它已追平并重新成为正常链成员。

## 恢复过程应该怎么做

最实用的方式不是“直接全盘复制”，而是：

1. 先拿恢复节点的 chunk metadata 列表。
2. 再拿前驱的 chunk metadata 列表。
3. 比较两边的版本和存在性。
4. 只传输缺失或落后的块。

这样做的好处是：

- 恢复不必全量复制。
- 可和正常写流并行存在。

## 为什么恢复路径必须和正常写路径协同

如果恢复过程中正常写还在发生，就不能简单地“先拷完再开写”。更合理的方式是：

- 正常写继续走主链。
- 对 `syncing` 节点，必要时把增量写升级成全量块写。
- 最后用一次 `sync_done` 确认它已经追平。

这也是为什么恢复状态机必须参与写转发逻辑。

## 自己重做时推荐的取舍

- 第一版先手工恢复，但状态机名字和状态流先定义好。
- 第二版先做“落后节点恢复”，不要一开始就做最复杂的多种故障并发恢复。
- 恢复节点的可见性要谨慎，默认不要让客户端直接读它。

## 常见坑

### 坑 1：只有 online/offline 两种状态

这样恢复过程要么暴露给客户端太早，要么完全没地方表达“正在恢复中”。

### 坑 2：恢复逻辑完全脱离正常写路径

一旦恢复期间还存在正常写，这种设计几乎一定会出现版本分叉或追平失败。
