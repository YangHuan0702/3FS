# 原生高性能 IO 路径

## 为什么需要它

FUSE 的价值是兼容性，不是极致性能。对 3FS 这类面向训练和推理场景的系统来说，高性能路径通常必须额外设计。

目标是：

- 减少内核与用户态之间的数据拷贝。
- 减少 syscall 次数。
- 用 batch I/O 提高 storage 请求效率。

## MVP 怎么看待这个问题

第一版完全可以不实现高性能路径，但接口设计上建议预留两件事：

- 文件描述符注册能力。
- 客户端能管理一组共享内存 buffer。

这样以后从 FUSE 演进到共享内存 + ring buffer 时，不至于推翻客户端模型。

## 推荐的实现方向

最自然的路线是：

1. 用户进程通过 API 创建共享内存大 buffer。
2. 用户进程通过 API 创建一个或多个 I/O ring。
3. FUSE 进程作为消费者，从 ring 中批量取请求。
4. 由 storage client 聚合并执行这些请求。

这条路线本质上是在用户态复刻“类似 io_uring 的提交/完成模式”。

## 关键数据结构应该有哪些

- `Iov`
  共享数据 buffer。
- `Ior`
  共享请求 ring。
- `Sqe/Cqe`
  提交和完成项。
- `registered fd`
  文件描述符到内部 inode/session 的注册映射。

## 为什么它比“直接加线程”更有效

如果只是给 FUSE 多开线程，你依然绕不过：

- FUSE 队列本身的锁竞争。
- 内核与用户态复制。
- 小 I/O 请求拆分与调度开销。

而共享内存 + ring 模型的优势在于：

- 用户进程和 FUSE 进程直接共享数据 buffer。
- 多个小 I/O 可以自然批处理。
- 请求准备与请求执行可以分离。

## 设计时最关键的几个问题

### 一个 ring 是否允许多线程并发生产

建议一开始限制：

- 一个 ring 的准备路径单线程。
- 多 worker 可以并行消费不同 batch。

否则很容易把提交顺序和完成顺序管理搞复杂。

### 批处理策略怎么定

建议给 ring 一个 `io_depth` 和 `timeout`：

- 积累到一定数量就发批。
- 不足一批但超时也发。

这是高性能路径是否真的有效的关键。

### 写路径如何与元数据语义对齐

即使数据写走高性能路径，也不能丢掉：

- length hint
- truncate version
- close / sync 后的最终长度更新

所以高性能 I/O 只是数据面优化，不能自己发明一套完全脱离 meta 的语义系统。

## 向 3FS 风格演进时要补什么

- 按优先级和 ring 深度调度批量 I/O。
- 允许 read uncommitted 等特定读模式。
- 维护 buffer / ring 的 mount 级命名空间。
- 对读写执行做单独指标采集。

## 自己重做时推荐的取舍

- 第一版不必追求完全零拷贝，但共享内存模型要先定下来。
- 先做 `read` 的高性能路径，再做 `write`，因为写还会牵涉长度与会话语义。
- 把高性能 I/O 看成客户端子系统，不要散落到 FUSE 各个回调里。

## 常见坑

### 坑 1：把高性能 I/O 写成另一套“微型文件系统”

正确做法是：

- 元数据语义继续归 meta / FUSE 控制层。
- 高性能路径只优化数据提交与完成。

### 坑 2：没有明确的 batch 形成策略

如果只是把请求塞进 ring，却不定义何时形成 batch，性能收益通常很有限。
