# 配置观测与运维

## 这个模块解决什么问题

很多系统在原型阶段只关心“能不能读写”，但到真正可维护时，下面几件事会迅速变重要：

- 配置如何组织。
- 配置如何下发。
- 日志和指标怎么统一。
- 出故障时用什么 CLI 或脚本定位问题。

## MVP 怎么实现

### 配置建议分三层

- `实例配置`
  例如 node id、listen address。
- `启动配置`
  例如 cluster id、mgmtd 地址。
- `运行配置`
  例如线程池、超时、批处理大小、重试参数。

即使第一版都落在本地文件里，也建议逻辑上先分层。

### 最小运维入口

建议至少做一个 CLI，支持：

- 节点注册/查看
- 路由查看
- 创建目录/文件的基础操作
- 查询 chunk 或 target

这样你后面调试集群时不至于只剩下 grep 日志。

## 为什么真实工程会更复杂

### 配置不只是文件

一旦系统有 mgmtd，配置就会慢慢演进成：

- 模板
- 渲染
- 校验
- 热更新

而不仅仅是一份 TOML。

### 指标不是“以后再加”

存储系统出问题时最需要的是：

- 当前延迟
- 当前重试数
- 当前读写量
- 某条恢复路径是否卡住

如果这些都没有，就只能靠日志猜。

## 推荐的观测面拆分

### 日志

至少分：

- 普通运行日志
- 错误日志
- 致命日志

### 指标

至少覆盖：

- 请求耗时
- 请求次数
- 重试次数
- batch 大小
- 读写字节数
- 恢复相关状态数

### 调试接口

至少提供：

- dump routing info
- dump chain info
- query chunk meta
- dump inode / direntry

## 向 3FS 风格演进时要补什么

- 配置模板由 mgmtd 下发。
- 服务启动时先拿模板再渲染本地完整配置。
- 支持 `validate config` 和 `hot update config`。
- 独立 monitor collector 汇聚指标。

## 自己重做时推荐的取舍

- 第一版就把日志和指标接口抽成公共组件。
- 第一版 CLI 即使功能少，也要先建起来。
- 不要等到多机调试时才想到“需要 query chunk”。

## 常见坑

### 坑 1：所有参数全靠编译期常量

这样以后做性能调试时，会不断重复：

- 改代码
- 重编
- 重启集群

成本太高。

### 坑 2：指标只做总量，不做分路径

比如只看总 QPS 而不区分：

- 正常读写
- forward
- recovery
- metadata batch

这样定位问题会非常慢。
