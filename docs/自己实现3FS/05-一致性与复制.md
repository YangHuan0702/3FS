# 一致性与复制

## 这个模块解决什么问题

如果系统要从“能跑”变成“分布式存储”，一致性和复制就是核心。对 3FS 风格系统来说，重点不在于追求最复杂的共识协议，而在于：

- 数据写入有明确副本传播顺序。
- 客户端重试不会把数据写乱。
- 节点故障后副本关系仍可推演。

## MVP 怎么做

最小版可以完全不做复制，只做：

- 单副本写入。
- 请求携带 request id。
- chunk metadata 保存 version。

理由是：

- 复制逻辑可以后加。
- 但如果请求身份和版本字段不留，后面很难无痛升级。

## 如果要开始做复制，推荐什么路线

最推荐的是链式复制，而不是一上来做多数派写。原因是：

- 读写路径容易理解。
- 存储服务端实现更直接。
- 恢复逻辑与顺序传播天然匹配。

### 基本模型

- 一条 chain 由多个 target 组成。
- 写从 head 进入。
- 沿链向 tail 转发。
- tail 提交后回 ACK。
- ACK 反向传播，前序副本完成 committed 切换。

## 为什么要做 committed / pending 两阶段

如果写请求在中途失败，你必须区分：

- 已经写到本地但尚未全链确认的版本。
- 已经完成全链提交的版本。

这就是 `pending` 与 `committed` 的本质价值。没有这层区分：

- 恢复时无法判断该保留哪个版本。
- 读请求无法知道该不该返回当前数据。

## 幂等为什么一定要做

客户端看到超时后通常会重试。如果 storage 端不能识别重复请求，就会出现：

- 同一写被执行两次。
- sequence 被破坏。
- ACK 和 commit 版本不一致。

因此推荐设计：

- 每个 client 拥有 `client_id`
- 每个写通道拥有 `channel_id`
- 每个通道写请求按 `seqnum` 单调递增
- 每个请求还有 `request_id`

服务端缓存最近结果，重复请求直接回旧结果。

## 复制路径里需要哪些状态

建议至少保留：

- `chain_version`
- `chunk_version`
- `request_id`
- `channel_seqnum`
- `public_target_state`
- `local_target_state`

如果这些状态不显式建模，复制路径和恢复路径会混成一团。

## 向 3FS 风格演进时要补什么

### 演进 1：允许链版本变化

现实里 successor 可能在写传播过程中失效。正确做法不是让客户端解决一切，而是：

- mgmtd 更新 chain version。
- 上游节点重新查路由后重试转发。

### 演进 2：恢复中的节点要能接收全量块

处于 `syncing` 的节点不应只接收 patch，而应能接受完整 chunk 替换写。否则无法保证恢复后状态一致。

### 演进 3：读请求要知道自己能读什么版本

当本地同时存在 committed 和 pending 版本时，读路径需要明确定义：

- 普通读默认返回 committed。
- 某些恢复或调试路径可以允许读 pending。

## 自己重做时推荐的取舍

- 第一版不做复制，但把字段和接口先留出来。
- 第二版优先做链式复制，而不是更复杂的一致性协议。
- 重复请求缓存一定下沉到 storage 服务端，而不是只靠客户端去避免。

## 常见坑

### 坑 1：把 request id 当日志字段而不是协议字段

如果 request id 不参与存储端判定，那它在超时重试场景里几乎没价值。

### 坑 2：直接原地覆盖 committed

这样在中途失败时几乎没有恢复空间。推荐始终保留“新的未提交版本”和“旧的已提交版本”的区别。
