# 存储服务

## 这个模块解决什么问题

存储服务负责管理真实的数据块及其本地持久化。它不关心目录树，但必须关心：

- chunk 的读写。
- chunk 元数据。
- 物理位置分配。
- 本地 target 的健康状态。

## MVP 怎么实现

### 基本单位：chunk

建议一开始就使用固定大小 chunk：

- 文件按照 `chunk_size` 切分。
- `chunk_id = inode_id + chunk_index`。
- 每个 chunk 拥有自己的元信息。

### 本地持久化建议

最小版有两种可接受方案：

- 方案 A：数据和元数据都先放本地文件系统。
- 方案 B：数据块放数据文件，元数据放 RocksDB/SQLite。

推荐方案 B，因为后续扩展更自然。

### 存储服务最小接口

- `read_chunk`
- `write_chunk`
- `truncate_chunks`
- `remove_chunks`
- `query_chunk_meta`

这些接口就足够支撑第一版 FUSE 和 meta。

## 本地存储的推荐结构

### 数据文件

建议不要为每个 chunk 单独建文件。更现实的方式是：

- 每块盘维护若干大的 data file。
- chunk 写入其中的某个 offset。

### 元数据

至少记录：

- `chunk_id`
- `target_id`
- `offset`
- `length`
- `checksum`
- `version`
- `timestamp`

这样后续要做 query、恢复和复制时，不会完全靠文件名推断状态。

## 为什么真实工程会更复杂

### 存储不只是“读文件写文件”

一旦进入多副本和恢复阶段，storage 还必须支持：

- pending / committed 两阶段版本。
- query 某个 chunk 的版本状态。
- 导出全量 chunk metadata。
- 接收整块替换写。

### 空间分配和元数据落盘不能揉成一坨

更稳定的设计是：

- 先由分配器给出新的物理位置。
- 再完成数据写入。
- 最后原子落下 chunk metadata。

这也是为什么“块引擎”在真实系统中往往会成为一个独立子系统。

## 向 3FS 风格演进时该怎么做

### 演进 1：target 作为一等对象

不要把 storage 只建模成“一个节点一个目录”。建议尽早引入：

- `target`
- `disk_index`
- `target_state`

这能让未来的恢复和摘链以 target 为单位发生。

### 演进 2：更新执行按磁盘切队列

如果所有写都塞进一个全局队列，系统在压力大时很难稳定。更实用的方式是：

- 每个磁盘或 target 分配自己的队列。
- worker 按磁盘并行处理。

### 演进 3：把 chunk 元数据查询做成正式能力

后面恢复和调试一定会依赖：

- 扫描某个 target 上所有 chunk。
- 查询某个 chunk 当前 committed / pending 版本。

因此一开始就别把这些能力藏在内部。

## 自己重做时推荐的取舍

- 第一版先不做自研块引擎，可以用简单的数据文件 + 元数据 KV。
- 校验和从第一版就加上。
- 先不做极致空间回收，但要让“chunk 元数据”和“物理块位置”有稳定映射。

## 常见坑

### 坑 1：一个 chunk 一个文件

小系统阶段看起来方便，但会快速遇到：

- inode 数量爆炸
- 目录扫描慢
- 删除和恢复成本高

### 坑 2：不记录版本

即使单副本阶段，也建议先把 `version` 字段放进 chunk 元数据里。后面复制和恢复几乎一定会用到。
