# 系统总览与 MVP 边界

## 要实现的到底是什么

如果你要做一个 3FS 风格系统，本质上是在做一个面向高吞吐共享访问场景的分布式文件系统。它和对象存储或普通 NAS 的差异主要在：

- 提供真正的文件系统语义，而不是 key-value 风格的对象操作。
- 元数据和数据面显式分离。
- 客户端获取布局后，数据读写尽量绕过 meta 直达 storage。
- 后期可以支持比普通 FUSE 更高性能的 I/O 路径。

## 最小可运行版应当包含什么

第一版建议只做下面这些。

### 必须有的

- 目录、文件、符号链接的最小语义。
- inode 与目录项的元数据模型。
- 一个中心化的 mgmtd，负责注册和配置。
- 一个 meta 服务，负责文件系统操作。
- 一个 storage 服务，负责 chunk 读写。
- 一个 FUSE 客户端，把系统挂载出来。

### 暂时不要做的

- 多主 mgmtd。
- 多副本一致性。
- 热更新配置。
- 零拷贝原生 I/O。
- 存储分层和复杂 compaction。
- 全自动恢复流程。

## 推荐的初始架构

第一版建议就采用 4 角色结构：

- `mgmtd`
  负责节点注册、基础路由和配置源。
- `meta`
  负责文件系统语义。
- `storage`
  负责按 chunk 读写数据。
- `fuse client`
  负责提供 POSIX 接口。

这样做的原因不是为了“看起来像 3FS”，而是因为这个边界本身就是合理的。后面无论要加复制、恢复还是高性能 I/O，都要依赖这条边界。

## MVP 的数据模型

### 元数据

建议从下面几个结构开始：

- `Inode`
  保存文件类型、权限、属主、时间戳、文件长度、布局。
- `DirEntry`
  保存父目录 inode id 与名字到子 inode id 的映射。
- `Layout`
  保存 chunk 大小、stripe 大小、目标 chain 或 target 列表。

### 数据

建议采用固定大小 chunk：

- 文件按 `chunk_size` 切分。
- `chunk_id = inode_id + chunk_index`。
- 每个 chunk 在 storage 上有独立元信息和物理位置。

这样可以极大降低后续复制、恢复、截断和 GC 的复杂度。

## MVP 的调用路径

### 元数据操作

例如 `mkdir/create/rename/remove/stat/list`：

1. FUSE 客户端把请求发给 meta。
2. meta 在事务里更新 inode / direntry。
3. 返回更新后的 inode 或状态码。

### 数据读写

例如 `read/write`：

1. 客户端先通过 `open` 拿到 inode 和 layout。
2. 根据偏移计算 chunk id。
3. 直接向 storage 发读写请求。
4. 必要时回 meta 更新长度、时间戳或 close/sync 状态。

这个路径里最重要的思想是：

- meta 不应该成为每次数据 I/O 的中转站。

## 第一版最容易做错的地方

### 错误 1：把 meta 和 storage 融在一个进程里

这样短期看很快，长期几乎一定重构。原因是：

- 元数据语义和数据 I/O 的热点完全不同。
- 后面复制和恢复的复杂度会把这个单体拖垮。

### 错误 2：用变长块或对象切片代替固定 chunk

固定 chunk 不是唯一选择，但它是最适合先做对的选择。因为：

- 长度查询更简单。
- 副本复制更简单。
- 故障恢复更简单。
- chunk 范围扫描更简单。

### 错误 3：把 FUSE 当成最终性能路径

FUSE 非常适合作为第一版接入方式，但不要把整体性能模型绑死在 FUSE 的线程队列和内核拷贝路径上。第一版就应该预留“未来存在独立高性能 I/O 路径”的接口空间。

## 如果参照 3FS，后面会怎么演进

从 MVP 往 3FS 风格演进，通常是这条路线：

1. 单副本 storage -> 链式复制。
2. 基础请求处理 -> 带幂等和重复写缓存的 update 路径。
3. 普通 FUSE -> 用户态共享内存 + I/O ring。
4. 本地静态配置 -> mgmtd 模板配置和动态更新。
5. 手工恢复 -> 状态机驱动的恢复流程。

## 自己重做时的建议

- 第一版就定义好 `mgmtd/meta/storage/client` 四种协议边界。
- 即使单机部署，也按分布式服务的方式写。
- 把布局信息作为元数据中的一等对象，而不是藏在客户端逻辑里。
- 所有写请求从第一天开始就带请求身份字段，为后续幂等做铺垫。
