# 自己实现 3FS：阅读顺序与总览

## 这套文档写给谁

这套文档写给 C++ / 存储方向工程师。默认读者具备以下背景：

- 理解线程、锁、协程、异步 I/O、RDMA 这类基本概念。
- 理解文件系统的 inode、目录项、页缓存/写回、元数据与数据面分离等常见设计。
- 理解分布式系统的副本、一致性、租约、路由信息和故障恢复。

这套文档不追求逐文件解释当前 3FS 仓库，而是回答一个更直接的问题：

如果你要自己实现一个 3FS 风格的分布式文件系统，应该按什么顺序想问题，哪些地方先做简化版，哪些地方必须从一开始就设计对。

## 这套文档的写法

每篇都会按相同结构组织：

- 这个模块解决什么问题。
- 最小可运行版应该怎么做。
- 为什么真实工程要比 MVP 复杂。
- 如果参照 3FS 风格演进，应该往哪个方向走。
- 重做时建议保留哪些简化，避免哪些坑。

## 推荐的实现阶段

如果你要自己从零写一个 3FS 风格系统，推荐拆成 5 个阶段。

### 阶段 1：跑通最小闭环

目标是得到一个“能挂载、能建目录、能创建文件、能读写”的系统。此时建议：

- 只做单副本存储。
- 元数据只做最小事务语义。
- 客户端只做 FUSE。
- 控制面只做单主 mgmtd。

### 阶段 2：把控制面和数据面拆干净

这个阶段开始把系统从“原型”拉向“架构化”：

- mgmtd 负责路由和配置。
- meta 只负责语义。
- storage 只负责块数据。
- 客户端只拿布局后直接找 storage。

### 阶段 3：引入复制和幂等

这是从单机原型变成分布式存储系统的关键点：

- 引入 chain replication 或近似方案。
- 给写请求加上 request id / channel seq。
- 让 storage 能识别重复写和超时重试。

### 阶段 4：引入恢复和状态机

这一步的重点不是提速，而是让系统可恢复：

- 目标节点下线后如何摘链。
- 节点回来后如何进入 `waiting/syncing/serving`。
- 正常写流与恢复流如何共存。

### 阶段 5：引入高性能客户端路径

FUSE 先解决易用性，高性能路径再单独做：

- 共享内存 buffer。
- 用户态 I/O ring。
- 批量读写、零拷贝和更少的内核交互。

## 实现时最重要的三条原则

### 原则 1：先把边界画对，再补性能

很多存储系统在第一版就过度追求极致性能，导致控制面、元数据面和数据面纠缠在一起。正确顺序应该是：

- 先把职责边界切清楚。
- 再把关键一致性点补上。
- 最后针对热点路径做性能专用优化。

### 原则 2：MVP 可以简化功能，但不要破坏演进方向

可以简化：

- 单主 mgmtd
- 单 meta
- 单副本 storage
- 不做热更新
- 不做零拷贝 I/O

不要破坏：

- 控制面与元数据面分离
- 元数据与数据路径分离
- 布局信息能让客户端直达 storage
- 写请求具备幂等基础

### 原则 3：把“恢复”当成一等公民

分布式文件系统里最容易被低估的不是读写主路径，而是恢复路径。一个建议是：

- 第一版可以不自动恢复，但接口和状态机一定要先留出来。

否则后面加恢复时，几乎所有读写路径都会被推倒重来。
