# 测试策略与演进路线

## 为什么最后要单独写这篇

一个分布式文件系统能不能长期演进，不只看架构是否优雅，还看：

- 有没有可靠的测试分层。
- 每个阶段的功能边界是否清晰。
- 有没有把正确性验证前置。

## 推荐的测试分层

### 第 1 层：基础单元测试

优先覆盖：

- 配置解析
- inode / direntry 编解码
- chunk id 和布局计算
- 请求 id / 通道序号处理
- 状态机转移

### 第 2 层：模块测试

分别测试：

- mgmtd
- meta
- storage
- client

例如：

- `rename` 是否原子。
- 重复 update 是否返回缓存结果。
- routing version 变更后 client 是否能刷新。

### 第 3 层：最小集群集成测试

这是最重要的一层之一。建议用脚本自动拉起：

- 1 个 mgmtd
- 1 个 meta
- 1~2 个 storage
- 1 个 FUSE client

然后从挂载点做真实读写。

### 第 4 层：故障测试

至少覆盖：

- 杀掉 storage 后读写是否按预期失败或切换。
- returning target 是否能恢复。
- 重试中的请求会不会重复写。

### 第 5 层：协议/状态机验证

如果条件允许，推荐把复制和恢复状态机用形式化规格工具验证。哪怕不是完整证明，也能帮你提前发现状态遗漏。

## 推荐的演进路线

### 阶段 A：单机语义正确

目标：

- 完成 inode、目录项、基础 FUSE 操作。
- 数据单副本写到一个 storage。

验收标准：

- 能挂载。
- 能跑基础文件操作。
- 重启后元数据和数据不丢。

### 阶段 B：控制面独立

目标：

- 引入 mgmtd。
- client 通过 mgmtd 获取路由和配置。

验收标准：

- 服务不再依赖手写固定地址。
- 配置与节点注册可通过管理入口查询。

### 阶段 C：单副本分布式闭环

目标：

- meta 与 storage 分离部署。
- client 可根据 layout 直达 storage。

验收标准：

- 大多数数据 I/O 不经 meta 转发。
- truncate / close / sync 的长度语义正确。

### 阶段 D：复制与幂等

目标：

- 引入 chain replication。
- 完成 request id / channel seq 幂等保护。

验收标准：

- 写请求全链传播。
- 重试不会造成重复写。

### 阶段 E：恢复与状态机

目标：

- target offline 后自动摘链。
- 节点恢复后可重新同步并 serving。

验收标准：

- 恢复过程中系统仍能继续对外提供受限服务。

### 阶段 F：高性能客户端路径

目标：

- 共享内存 buffer。
- 用户态 I/O ring。
- 批量读写。

验收标准：

- 小 I/O 或高并发随机读吞吐明显优于纯 FUSE 路径。

## 自己重做时推荐的测试取舍

- 每完成一个阶段就冻结一批集成测试，不要所有功能做完再补。
- 恢复和幂等的测试要早于性能压测。
- 先测“错误时是否安全”，再测“正常时是否更快”。

## 常见坑

### 坑 1：只做 happy path 测试

分布式文件系统最容易出问题的地方通常是：

- 超时后重试
- 节点半恢复状态
- 路由版本切换
- close/sync/truncate 交错

### 坑 2：压测跑得动就以为设计成立

如果没有故障测试和状态机测试，压测结果往往只能说明：

- 在理想环境下当前版本没立即崩。

不能说明：

- 数据一定一致。
- 恢复一定正确。

## 最后建议

把“自己实现 3FS”这件事当成一个分阶段、可回退、可验证的工程，而不是一次性写完的宏大项目。真正可靠的路线通常是：

- 先做边界正确的简化版。
- 再逐步把复杂性引入到正确的位置。
